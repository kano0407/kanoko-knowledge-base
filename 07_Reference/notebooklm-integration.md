# NotebookLM-py vs 現在のメモリシステム（MEMORY.md）

## 概要比較表

| 側面 | NotebookLM-py | MEMORY.md（現在） |
|---|---|---|
| **用途** | 大量ドキュメント分析 | セッション間の文脈保持 |
| **トークン消費** | ほぼ0（Gemini処理） | セッション開始時のロードのみ |
| **対応ファイル数** | 30+ドキュメント | 200行以下推奨 |
| **分析形式** | 自動生成（ポッドキャスト、要約、など） | 手動記述 |
| **永続性** | Googleアカウント（NotebookLM側） | ローカルファイル（claude/.claude/projects/） |
| **セッション管理** | 独立（実行ごとに新規） | 自動ロード（毎回） |

## 各システムの得意領域

### NotebookLM-py が得意なこと
- ✅ 複数のYouTube動画・PDF・URLから大量情報を抽出
- ✅ 話題・企画・戦略のアイデア出し（「これらの記事から3つのポッドキャスト台本を生成」）
- ✅ トークン消費なしに複雑な分析タスク実行
- ✅ ハルシネーション少ない（Gemini処理）
- ✅ MP3・MP4・JSON・CSVなど多形式エクスポート

### MEMORY.md が得意なこと
- ✅ セッションをまたいだ文脈管理（「以前決めたルールを今回も適用」）
- ✅ ユーザーの意思決定・フィードバック記録
- ✅ 素早い参照（毎回自動ロード）
- ✅ プロジェクト状態の変化を時系列で追跡
- ✅ 複数セッション間での一貫性維持

## 推奨される使い分けパターン

### パターンA: 大量ソース分析 → 結果をMEMORYに保存
```
1. YouTube配信10本 + 業界記事5本をNotebookLMに投入
2. NotebookLMが「今月のトレンド分析」「おすすめ企画3案」を自動生成
3. その結果（出力）をMEMORY.mdの project_topic.md に保存
4. 次回セッション開始時、MEMORY.mdから戦略を参照
```

### パターンB: 既知の文脈を素早く復旧
```
1. 前回セッション：「X運用はブックマーク重視」「台本は共創型」をMEMORY.mdに記録
2. 今回セッション：MEMORY.mdが自動ロード
3. 説明不要で「前回決めた方針で」と指示可能
```

### パターンC: 継続的な企画開発
```
1. 企画案：NotebookLMで記事群から生成
2. 決定内容：MEMORY.mdに「3月採用企画」として記録
3. 実装詳細：tech/CLAUDE.md や contents/CLAUDE.md で実装
4. 実績：次月、新しいNotebookLM分析に前月実績をソースとして追加
```

## 実装上の注意点

### NotebookLM-pyの制約
- ⚠️ 非公式API（Googleが予告なく変更する可能性）
- ⚠️ ブラウザログイン必須（初回は人間がログイン、以降はtoken保存）
- ⚠️ NotebookLM側で一定数のノートブックしか保持不可（定期削除推奨）

### MEMORY.md の制約
- ⚠️ 200行超え → 自動トリミング（優先度付けが必須）
- ⚠️ 手動管理（ユーザーが不要な行を削除しない限り増殖）
- ⚠️ 毎回ロードされるため、常に最新のコンテキストを含むべき
